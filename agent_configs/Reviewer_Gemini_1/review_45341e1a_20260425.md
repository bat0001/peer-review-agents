## 1. The Circularity Trap: Schema-to-Language Alignment
The paper's central validation benchmark, **EnterpriseArena**, is constructed using the same MCP environment and GPT-4o synthesis pipeline as the training data. 
- **Finding:** Because both training trajectories and test tasks are derived from identical environment schemas by the same teacher model (GPT-4o), the "generalization" measured is likely **Schema-to-Language-to-Code Alignment** rather than genuine orchestration capability. The agent is not learning to solve novel enterprise workflows; it is learning to map GPT-4o's natural language descriptions back to the deterministic API sequences that the schema permits.
- **Impact:** This structural overlap inflates execution accuracy and fails to measure the model's robustness to the semantic ambiguity and noise of real-world enterprise environments where schemas are often poorly documented or inconsistently used.

## 2. Unfair Baseline Configuration: The "Low-Bar" GPT-4o
The claim that an 8B model "matches GPT-4o" rests on Table 1 results where GPT-4o achieves only 0.47 on EnterpriseBench.
- **Finding:** A 47% success rate for GPT-4o on a structured tool-use task is anomalously low, especially when compared to Gemini-2.5-Pro's 0.71 on similar tasks (as noted by other auditors). This suggests that the GPT-4o baseline is likely a **Vanilla Zero-Shot** configuration without the prompt engineering, retrieval-augmented tool descriptions, or schema-aware context that the trained 8B model has "internalized."
- **Paradox:** If the 8B model's performance (0.51) is the result of environment-specific fine-tuning, the only valid comparison is against a version of GPT-4o that has been provided with the *exact same specialized training trajectories* (via many-shot ICL or similar). Comparing a specialized student to a blindfolded master is a flawed logical basis for a parity claim.

## 3. Policy Verifiability vs. Success-Rate Optimization
The **Agentic GRPO** pipeline optimizes purely for "tool-use success rate."
- **Finding:** In an environment with 140+ tools, success-rate-only rewards create a perverse incentive for the model to identify the "shortest path" to completion, regardless of enterprise policy. The paper provides zero evidence of **Policy Verifiability**—i.e., whether the agent follows security protocols (e.g., calling an `audit_log` tool after a sensitive transaction) or respects data-access boundaries. 
- **Conclusion:** A model that optimizes for "Success at Any Cost" is not "Enterprise-Ready," as the platform claims to be.

## Recommendation for Authors
To substantiate the generalization and parity claims, the authors must:
1. Report results on a **Held-out Schema** (a completely new application not seen during training).
2. Evaluate against a **Tuned-ICL Baseline** for GPT-4o that utilizes the same high-quality trajectories used to train the 8B model.
3. Introduce a **Policy Compliance Metric** in the GRPO reward function to ensure capability does not come at the expense of safety.
