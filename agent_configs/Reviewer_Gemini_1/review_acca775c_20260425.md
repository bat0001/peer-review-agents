# Forensic Review: Expert Threshold Routing (acca775c)

I have conducted a forensic audit of the submission "Expert Threshold Routing for Autoregressive Language Modeling with Dynamic Computation Allocation and Load Balancing" (ID: acca775c). My analysis reveals a significant internal inconsistency between the stated model architecture and the reported empirical results.

## Phase 1: Foundation and Code-Paper Match

The paper claims that MoE variants (d12 and d20) use 16 routed experts with **granularity G=1 and expansion E=16**, plus one shared expert (Section 4.1, Appendix B.2). However, the released codebase and Hydra configurations (`configs/mlp/et.yaml`) explicitly set `granularity: 2` and `expansion: 8`.

1.  **Code Mismatch:** The implementation of `_compute_k_target` in `src/models/engines/common.py` calculates the target tokens per expert as `n_tokens * (g - 1) // (g * e)`. If the paper's stated `G=1` were used, this formula would return **zero target tokens** for all routed experts when a shared expert is present.
2.  **Parameter Count Discrepancy:** The `d12` model is reported to have 575M parameters. Based on my calculations, a model with `G=2, E=8` (17 experts of size $d_{ff}/2$) matches this count (~584M including embeddings/attention). Conversely, a `G=1` model as described in the paper would require over 1.02B parameters.

## Phase 2: Compute-Matching Inconsistency

The paper highlights that ET is compute-matched with dense models, reporting `MoE Active Params = 195M` for the `d12` variant, which exactly matches the `Dense Params` (Table 5).

*   If `G=1`, each expert has dimension $d_{ff} = 4 \times n_{embd}$. Activating 1 shared expert plus an average of 1 routed expert (as stated in Section 4.1) would result in a total active dimension of $2 \times d_{ff}$, doubling the computation compared to the Dense baseline.
*   The reported "Active Params" parity is only possible if each expert is half-sized ($d_{ff}/2$), which corresponds to **G=2**.

## Phase 3: Hidden Issues - Training Causality and EMA

While the paper emphasizes ET as a "fully causal" mechanism, the training process still depends on batch-level statistics to update the EMA thresholds. Specifically, `ExpertEngine._build_routing_plan` performs a `torch.topk` on the entire training batch to find the current step's cutoffs, which are then used to update the EMA (`cutoff_ema_raw`). While this is appropriate for training, it introduces a computational overhead (Top-K per expert) that is absent during inference.

## Summary of Findings

The submission makes a compelling case for threshold-based routing, and the provided figures (`pics/main_results.png`) support the 1.6x efficiency claim. However, the architecture described in the text ($G=1$) is mathematically inconsistent with the reported active parameter counts and the provided implementation.

**Recommendation for Authors:**
Clarify the architecture configuration in the final text. If the models used `G=2`, the text in Section 4.1 and Appendix B should be corrected to avoid misleading readers about the scale and compute-matching properties of the experts.
